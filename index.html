IO模型

发表于 2021-04-08

socket在创建时默认是阻塞的。可以通过给socket系统调用的第二个参数传递SOCK_NONBLOCK标志，或者通过fcntl系统调用的F_SETFL命令，将其设置为非阻塞的。阻塞和非阻塞概念可以应用到所有文件描述符，而不只是socket

阻塞IO

针对阻塞IO执行的系统调用可能因为无法立即完成而被操作系统挂起，直到等待的时间发生为止。

例如客户端通过connect向服务器发起连接，connect将首先发送同步报文段给服务器，然后等待服务器返回确认报文段。如果服务器的确认报文段没有立即到达客户端，则connect调用将被挂起，直到客户端收到确认报文段并唤醒connect调用。

socket的基础API中，可能被阻塞的系统调用包括accept、send、recv和connect

非阻塞IO

针对非阻塞IO执行的系统调用总是立即返回，而不管事件是否已经发生。如果事件没有立即发生，这些系统调用就返回-1，和出错的情况一样。所以非阻塞IO需要根据errno来区分这两种情况。对于accept、send和recv来说，事件未发生时errno通常被设置为EAGAIN或者EWOULDBLOCK；对于connect而言，errno则被设置成EINPROGRESS。

显然，只有在事件已经发生的情况下操作非阻塞IO才能提高程序的效率。因此，非阻塞IO通常要和其他IO通知机制一起使用，比如IO复用和SIGIO信号

IO复用

IO复用是最常使用的IO通知机制，它指的是应用程序通过IO向内核注册一组事件，内核通过IO复用函数把其中就绪的事件通知给应用程序。常用的IO复用函数：select，poll和epoll

IO复用函数本身是阻塞的，它们能提高应用程序效率的主要原因在于他们具有同时监听多个IO事件的能力。

SIGIO信号

SIGIO信号也可以用来报告IO事件，可以为一个目标文件描述符指定宿主进程，被指定的宿主进程将捕获到SIGIO信号，这样当目标文件描述符上有事件发生时，SIGIO信号的信号处理函数将被触发，也就可以在该信号处理函数中对目标文件描述符执行非阻塞IO操作。

小结与对比

  • 理论上说，阻塞IO、IO复用和信号驱动IO都是同步IO模型。因为这三种IO模型中，IO的读写操作都是IO事件发生之后，有应用程序完成的。

  • 对异步IO而言，用户可以直接对IO执行读写操作，这些操作高速内核用户读写缓冲区的位置，以及IO操作完成之后内核通知应用程序的方式，异步IO的读写操作总是立即返回，而不论IO是否是阻塞的，因为真正的读写操作已经有内核接管。

  • 也就是说，同步IO模型要求用户代码自行执行IO操作，而异步IO机制则由内核来执行IO操作。

  • 同步IO向应用程序通知的是IO就绪事件，异步IO向应用程序通知的是IO完成事件

IO模型 读写操作 阻塞阶段
阻塞IO 程序负责控制IO读写 程序阻塞于函数读写
IO复用 程序阻塞于IO复用系统调用，但是系统调用可以监听多个IO事件 对IO本身的读写操作是非阻塞的
SIGIO信号 信号触发读写就绪事件，应用程序执行读写操作 程序没有阻塞阶段
异步IO 内核执行读写操作并触发读写完成事件 程序没有阻塞阶段

I/O复用函数的比较

三种IO复用函数的比较

select、poll和epoll三组IO复用系统调用都能同时监听多个文件描述符。它们会等待timeout指定的超过时间，直到一个或者多个文件描述符上有事件发生时返回，返回值是就绪的文件描述符的数量。返回0表示每有事件发生。

三个函数都通过某种结构体变量来高速内核监听那些文件描述符上的事件，并使用该结构体来获取内核处理的结果。

select

参数类型fd_set没有将文件描述符和事件绑定，它仅仅是一个文件描述符的集合，因此select需要提供3个这种类型的参数来分别传入和输出可读可写及异常事件，这一方面使得select不能处理更多类型的事件，另一方面由于内核对fd_set集合的在线修改，应用程序下次调用select前不得不重置这三个fd_set集合。

select存在允许监听的最大文件描述符数量上限，虽然可修改，但是可能会导致未定义行为出现。

poll

poll的参数类型pollfd把文件描述符和事件都定义其中，任何事件都被统一处理，使编程接口简洁的多。并且内核每次修改的都是pollfd结构体的revents成员，而events成员保持不变，因此下次调用poll时应用程序无需重置pollfd类型的事件集参数。

poll使用nfds参数指定最多监听的文件描述符和事件，可以达到系统允许打开的最大文件描述符数目——65535

select和poll相对来说更为相似

由于每次select和poll调用都返回整个用户注册的事件集合，所以应用程序索引就绪文件描述符的时间复杂度O(n)。同时，二者只能工作在相对低效的LT模式

epoll

  • epoll在内核中维护了一个事件表，并提供了一个独立的系统调用epoll_ctl来控制往其中添加、删除、修改事件。这样，每次epoll_wait调用都直接从该内核事件表取得用户注册的事件，从而无需反复从用户空间读入这些事件。
  • epoll_wait系统调用的events参数仅仅用来返回就绪的事件，这使得应用程序索引就绪文件描述符的时间复杂度达到O(1)。
  • 并且可以工作在高效的ET模式，并且可以设置EPOLLONESHOT来进一步减少事件触发次数
  • epoll_wait使用maxevents参数指定最多监听的文件描述符和事件，可以达到系统允许打开的最大文件描述符数目——65535
  • epoll_wait使用回调方式，内核检测到就绪的文件描述符时，触发回调函数，将文件描述符对应的事件插入内核就绪事件队列，最后在适当时机将就绪事件队列拷贝到用户空间。因此无需轮询
  • 注意！ 当活动连接比较多时，epoll_wait效率未必比select和poll高，因为此时回调函数会被触发的过于频繁。所以epoll_wait适用于连接数量多，但活动连接较少的情况

总结

系统调用 select poll epoll
事件集合 用户通过3个参数传入感兴趣的可读可写、异常事件，内核通过对这些参数在线修改来反馈其中就绪事件，用户每次调用select都要重置3个参数 统一处理所有事件类型，因此只有一个事件集参数。用户通过pollfd.events传入感兴趣事件，内核通过修改pollfd.revents反馈其中就绪的事件 内核通过一个事件表直接管理用户感兴趣的所有事件。因此用户无需反复传入所有感兴趣的事件。epoll_wait系统调用的参数events仅用于反馈就绪的事件。

50.合理的替换new和delete

替换的常见理由：

  1. 用来检测运用上的错误，可以在里面自己加写日志操作
  2. 为了强化效能，因为标准的new和delete需要处理所有的情况，所以对于特定的程序来说，它们可能会有额外的时空开销

四种类型转换：

  • const_cast:通常用来移除const，是c++唯一有此能力的类型转换操作符
  • dynamic_cast:主要用来“安全项下转型”，唯一无法有旧式语法执行的动作。会消耗大量的成本

29异常安全

异常安全函数提供下述三个保证之一：

  1. 基本承诺：如果异常抛出，程序内任何事物仍保持在有效状态下。没有任何对象或数据结构会因此而败坏，所有对象都处于一种内部前后一致的状态。但程序的现存状态并不能预测。即1->2的过程中抛出异常，有可能返回状态1，也有可能到达一个默认状态。
  2. 强烈保证：原子性，如果异常被抛出，程序状态不改变。调用这样的函数如果成功就是完全成功，如果失败就会恢复到调用之前的状态。
  3. 不抛掷(nothrow)保证：承诺绝不抛出异常！作用于内置类型上的所有操作都提供nothrow保证。

38 has-a关系

  • 复合的含义与public继承完全不同
  • 在应用域，复合意味着has-a；在实现域，复合意味着is-implemented-in-terms-of（以xx实现出）

针对第二点举个例子：比如要实现一个自己的set集合类，为了节省空间考虑通过list实现，显然不能用public继承，因为public继承意味着is-a，但是set并不包含重复元素，list是可包含的，所以说set is-a list是错误的，而应该是set包含一个list的成员。这就是is-implemented-in-terms-of。

39谨慎使用private继承

  • Private继承意味着is-implemented-in-terms-of。它通常比复合的级别低。但是当derived class需要访问protected base class成员时，或者需要重新定义继承而来的virtual函数时，这么设计是合理的。
  • 与复合不同，private继承可以造成empty base最优化。这对于“将对象空间最小化”的开发者来说可能很重要。

40多重继承和虚继承

关于virtual base class的忠告：

  • 非必要不使用virtual bases，平常就是用non-virtual继承。
    • 因为虚继承产生的对象会比non-virtual继承的对象体积大，访问速度变慢。这就代表着很多额外的开销，包括时间和空间；
    • 同时virtual base classes初始化的规则更加复杂且不直观：virtual base的初始化责任是由继承体系中的最低层class负责，也就暗示：（1）classes若派生字virtual bases而需要初始化，必须认知其virtual bases（2）当一个新的derived class加入继承体系内，它必须承担其virtual bases的初始化责任。
  • 如果必须使用virtual base classes，那么尽可能避免在其中放置数据，这样可以避免诡异的初始化和赋值操作。

Y和Z的大小受到三个因素的影响：

  1. 语言本身造成的额外负担：当语言支持virtual base classes就会导致一些额外负担，在derived class中，这是一个指针，指向virtual base class subobject或是一个相关的表格。表格中存放的是virtual base class subobject的地址或是其偏移位置。

  3. Alignment的限制：class Y和Z的大小目前为止5 byte，但大部分机器上聚合的结构体大小会受到alignment的限制，比如上述中alignment大小是4 byte，所以为了存取效率，将会被填充到8 byte。

  1. 4byte大小的指针，指向class X
  2. 1byte大小的char，因为Y是一个空类
  3. 3byte大小的alignment填充，凑齐4byte的整数倍

然后A的大小为12byte这是因为多重继承不会重复两个virtual base class X的部分。class A的大小由以下几点决定：

  1. 被共享的唯一一个class X实例，大小1byte
  2. Base class Y的大小，减去因为virtual base class X的大小，结果是4bytes，加上Base class Z的同样大小，同8bytes
  3. class A自己的大小：0byte
  4. class A的alignment大小，上述共9bytes，填充至12bytes
  5. 如果编译器针对empty virtual base class做了处理，则class X的1byte不再出现，也就不会填充，所以共8bytes

主码，超码，候选码，主属性

  • 超码：一个或多个属性，能够唯一标识一个元组。
  • 候选码：超码之中，任意真子集都不能构成超码的最小超码，称为候选码，简称码，一个表可以有多个码。
  • 主码：人为选中的候选码，作为唯一标识一个元组的候选码，称为主码。
  • 主属性：所有候选码的每个属性称为主属性

索引

索引通常使用B树或B+树实现，通过数据库系统维护的查找特定项的数据结构，实现高效查找。
优点：

  • 创建唯一性索引，可以保证数据的唯一性。
  • 加快数据检索速度。
  • 排序时也可加快速度。

缺点：

  • 创建索引和维护索引的时间开销。
  • 建立索引的空间开销。

B+树索引和Hash索引的区别

  • Hash可以通过特定运算，一次定位，检索效率高
  • Hash仅能满足“=”、“IN”、“”的查询，无法实现偏序等操作，无法进行排序。
  • 组合索引时，Hash索引不能利用部分索引进行查找。
  • 大量重复关键字Hash索引效率低下
  • B+树需要从根节点到叶节点多次IO

    视图和游标

  • 视图：
    SQL允许通过查询来定义”虚关系”, 在概念上包含查询的结果, 虚关系并不预先计算并存储, 而是在使用需关系的时候才通过执行查询倍计算出来。不是逻辑模型的一部分，但作为虚关系对用户可见的关系称为：视图。
    数据库会存储视图的定义本身，而不会存储视图的查询表达式的执行结果。也就是说，一旦视图关系出现在查询中，它就会被已存储的查询表达式代替。（有点像内联函数那感觉）

    视图的优点:

    1. 便于用户使用简单的查询语句进行查询
    2. 维护数据的独立性,视图可以从多个表中检索数据
    3. 便于权限的管理
    4. 相同数据可以产生不同的视图。

    视图的缺点：

    1. 相对于物化视图而言，效率较低，是类似内联函数，在执行视图查询时，DBS会将视图转换成SQL再进行查询。
    2. 大部分情况不允许对视图进行修改，会引起不一致情况出现。

  • 游标：
    SQL会把SQL作用结果暂时保存在内存中，可以使用游标来显示、操作这个结果的部分，比如第一条、第二条，前十条等。一般不使用，用于逐条处理数据时会使用。

存储过程

预编译的SQL语句，优点: 允许模块化设计，只需一次创建，可以多次使用。并且会预编译，执行效率较高。缺点:移植性差
存储过程与函数没有本质区别
存储过程 | 函数
-|-
用于完成特定的SQL语句操作 | 用于特定的数据
create procedure | create function
可以返回多个值 | 只有一个返回值
可作为独立语句执行 | 不可独立执行,必须作为表达式一部分
SQL语句不能调用存储过程 | SQL可以调用函数

drop、truncate、delete

  • drop:DDL删除掉整个表，释放所有空间，不能回滚
  • truncate:DDL不会触发触发器，删除表中所有数据，不会为单独的删除进行日志记录，不能回滚
  • delete:DML会触发触发器，使用where可每次删除一行，同时会保存每个删除通过日志保存用以回滚

临时表

临时表只会在当前连接中可见，当连接关闭时，SQL会自动删除并释放所有空间，不同链接中可以创建同名表。

五大范式

  1. 1NF:原子属性不可再分

  2. 2NF:非主属性完全依赖于主码。消除了非主属性对码的部分依赖
    例如：关系表为（学号，姓名，课程名称，成绩，学分）
    其中（学号，课程名）为主码
    （学号，课程名）$\rarr$（姓名，年龄，成绩，学分）
    而由于存在
    （学号）$\rarr$（姓名，年龄）
    （课程名称）$\rarr$(学分)
    所以该关系表不符合第二范式

  3. 3NF:消除了非主属性对码的传递依赖
    例如：存在关系表（#学号，学院，学院主任）
    其中（#学号）$\stackrel{f}{\rarr}$（学院）
    又有（学院）$\stackrel{f}{\rarr}$（主任）
    即存在非主属性对主码的传递依赖，故不符合第三范式。

  4. BCNF:3NF基础上删除了主属性对码的传递依赖和部分依赖。
    换句话说符合BCNF的关系模式中：

    • 对于任何$\alpha\rarr\beta$或者$\alpha$是R的一个超码
    • 这是一个平凡函数依赖($\beta\subseteq\alpha$)

    例如：存在关系模式（学生，教师，课程）其中每个教师只教授一门课程，每门课程可以由多名教师教授。某个学生选定一门课便对应一个固定的教师。
    可以得到如下函数依赖：
    （学生，课程）$\rarr$（教师）；
    （学生，教师）$\rarr$（课程）；
    （教师）$\rarr$（课程）。
    其中（学生，课程）和（学生，教师）都是候选码。该模型符合3NF因为没有任何非主属性对候选码的传递依赖或部分依赖。但不属于BCNF因为教师是决定因素而教师不是超码。

  5. 4NF：消除了多值依赖

TCP一些问题

为什么是3次握手而不是2次握手

三次握手具体：

  1. 客户端发给服务器SYN seq：确认服务器能建立连接
  2. 服务器发给客户端SYN ACK 并分配资源：确定服务器可以发送数据，可以建立连接，并且确认客户端的起始序列号
  3. 客户端第三次握手，并可以携带数据：确定客户端可以发送数据，并确认服务器端起始序列号

考虑这种情况，客户端发出SYN建立连接请求，之后该报文段超时，重发SYN建立连接，之后正常传输并关闭了连接。但是第一个超时报文段只是延时很久才到，并没有丢失，在关闭之后到达，这种情况下，服务器会进行连接的建立，但是客户端并没有数据要发送，导致了资源的浪费。

两次握手情况下，只有服务器对客户端的起始序列号做了确认，但客户端却没有对服务器的起始序列号做确认，不能保证传输的可靠性。

如何避免SYN DDos

  • 采用 SYN cookie 机制，服务器接收到SYN之后，并不知道是否是恶意攻击，因此不会打开半开连接，而是将一个秘密数，SYN源和目的IP地址与端口号进行散列函数，记录为cookie。服务器发送具有这种特殊初始序列号的SYNACK分组。服务器并不记忆该Cookie或任何对应于SYN的其他状态信息。
  • 如果用户合法，会返回一个ACK报文段。服务器接收到该segment，需要确认该ACK与之前的SYN是对应的。服务器会将之前发送的SYNACK segment中的源、目的IP地址和端口号以及秘密数进行相同的散列函数。如果该散列结果+1之后与客户的SYNACK确认值相同，则认为该连接合法。

为何四次挥手

因为建立连接时，服务器第二次握手只需回复SYN ACK而不需要携带实际内容。而在结束过程中，服务器收到客户端发来的结束segment之后，有可能还要发送许多消息，这些消息无法放在一个segment之中，所以需要四次挥手来保证服务器的所有消息传送完毕。

为何结束后提出结束方还需要等待两个MSL

MSL（segment最长生存时间 Maximum Segment Lifetime）：指segment在网络中最长生存时间，超出时间的segment会在接收后被丢弃。

  • 这可以保证，本次连接中的所有segment均已失活，不会出现本次连接中的segment出现在之后的网络环境中。
  • 试想这种情况：C发出了Fin segment，并且受到了对此确认，连接处于半关闭状态。之后S发出了所有segment之后也发送Fin segment，C收到之后对其作出ACK，并关闭了连接。但是最后一个ACK丢失了，此时S还处于未关闭状态，会不断在超时之后发送Fin segment，而C已经关闭了连接，不会再做出反应。导致S无法正常结束连接状态。

查询处理和查询优化

发表于 2020-04-20 更新于 2021-04-08

查询处理

查询处理是指从数据库中提取数据时涉及的一系列活动。这些活动包括：

  • 将高级数据库语言表示的查询语句翻译为能在文件系统的物理层上使用的表达式
  • 为优化查询而进行的各种转换，以及查询的实际执行。

概述

查询处理包括三个步骤：

  1. 语法分析与翻译
  2. 优化
  3. 执行

查询优化指：构造具有最小查询执行代价的查询执行计划，这是DBS的责任。

查询代价的度量

大型数据库中，最主要的代价是磁盘IO

表达式计算

物化

对关系代数表达式建立运算符树，之后自下而上开始运算，每次得到的结果都进行物化，建立起一个临时关系。这个临时关系还有可能需要写到磁盘中。

  • 某些情况下必须使用物化计算。
  • 物化计算：* 运算的没这种检结果被创建（物化），然后用于下一层的计算。

双缓冲： 使用两个缓冲区，一个用于连续执行算法，另一个用于写出结果。允许CPU和IO活动并行，从而提高算法执行速度。通过为输出缓冲区分配额外的磁盘块以及每次同时写出多个块，可以减少磁盘搜索速度。

流水线

通过减少查询中产生的临时文件数，可以提高查询效率。将多个关系操作组合成一个操作流水线实现，其中一个操作的结果传送到下一个操作，这样的计算称为流水线计算。

创建流水线的好处：

  1. 能够消除读和写临时关系的代价，减少查询计算代价。
  2. 如果一个查询计算计划的根操作符及其输入合并到流水线中，那么可以迅速开始产生查询结果。一旦他们生成，可以立即把结果显示给用户。

流水线可以按以下两种方式执行：

  1. 在一条需求驱动的流水线中，系统不停地向位于流水线顶端的操作发出需要元组的请求。每当一个操作收到需要元组的请求，它就计算下一个（若干个）元组并返回。该方式中，元组消极的按需产生。
  2. 生产者驱动流水线中，各操作并不等待元组请求，而是积极地产生元组。生产者驱动流水线中，各个操作作为系统中一个单独的进程或线程建模，以处理流水线输入的元组流，并产生相应的输出元组流。元组会放到输出缓冲区中，指导缓冲区已满，等待它的父操作将缓冲区中的元组移除。

关系代数运算的执行

选择运算

  • 线性搜索：慢，支持所有文件
  • 主索引，码属性等值比较
  • 主索引，非码属性等值比较
  • 辅助索引，等值比较

连接运算

  • 嵌套循环连接：代价最大，需要逐个检查两个关系中的每一对元组。
  • 块嵌套循环连接：内存缓冲区不能完全容纳一个关系式，使用块嵌套循环连接，以块的方式处理关系。在最坏情况下，对外层关系的每一个块读一次，而不是对外层关系的每个元组读一次。使用较小的关系作为外层关系更为有效，并且在内存中尽可能多的块用于存储外层关系的块
  • 索引嵌套循环连接：将元组数量较少的关系作为外层关系效果较好。
  • 归并连接：使用前最好对元组进行排序
  • 散列连接：用较少的关系作为构造用输入关系。

查询优化

查询优化：从许多策略中找出最有效的查询执行计划的一种处理过程。

查询执行计划的产生：

  1. 产生逻辑等价表达式
  2. 对所产生的表达式以不同的方式做注释，产生不同的查询计划
  3. 估计每个执行计划的代价，选择估计代价最小的一个

查询优化器中，上面3个步骤是交叉执行的：如产生一些表达式，加以注释后产生执行计划，之后再进一步产生一些表达式，再加以注释以此类推。

查询优化器使用启发式方法来减少优化代价：

  • 尽早执行选择运算
    例如：$\sigma_\theta(r\Join s)$其中$\theta$只引用$s$的属性。选择可以提前进行，然而如果$r$相对于$s$非常小，并且在$s$的连接属性上有索引，但在$\theta$属性的引用属性上无索引，那先做选择就需要在选择之后，对于$s$进行一次全体元组的扫描。这会加大代价。
  • 尽早执行投影运算（当投影涉及到了多个表中的属性时无法提前进行投影）

启发式算法并不验证采用本规则转换之后代价是否会减少，而是直接加以使用。

许多优化器允许为查询优化指定一个成本预算。当超过优化成本预算时停止搜索最有计划，返回当前的最优计划。

许多应用会反复执行同样的查询，不过查询中的常数不一样，例如经常会查询一个学生的所有信息，不同查询中只有学生ID不同，这种情况下，会将之前的最优计划进行缓存，缓存和重用查询计划成为计划缓存。

查询执行时间主要花费在慢速的磁盘IO上。
