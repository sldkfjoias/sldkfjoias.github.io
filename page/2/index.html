一些数据库相关问题

主码，超码，候选码，主属性

  • 超码：一个或多个属性，能够唯一标识一个元组。
  • 候选码：超码之中，任意真子集都不能构成超码的最小超码，称为候选码，简称码，一个表可以有多个码。
  • 主码：人为选中的候选码，作为唯一标识一个元组的候选码，称为主码。
  • 主属性：所有候选码的每个属性称为主属性

索引

索引通常使用B树或B+树实现，通过数据库系统维护的查找特定项的数据结构，实现高效查找。
优点：

  • 创建唯一性索引，可以保证数据的唯一性。
  • 加快数据检索速度。
  • 排序时也可加快速度。

缺点：

  • 创建索引和维护索引的时间开销。
  • 建立索引的空间开销。

B+树索引和Hash索引的区别

  • Hash可以通过特定运算，一次定位，检索效率高
  • Hash仅能满足“=”、“IN”、“”的查询，无法实现偏序等操作，无法进行排序。
  • 组合索引时，Hash索引不能利用部分索引进行查找。
  • 大量重复关键字Hash索引效率低下
  • B+树需要从根节点到叶节点多次IO

触发器trigger

用于保证完整性约束, 可以实现未被SQL约束机制制定的某些完整性约束。
设定条件：

  • 指明什么条件下执行触发器。被分解为一个引起触发器被监测的事件和一个触发器执行必须满足的条件。
  • 指明触发器执行时的动作。

何时不用触发器：

  • 在支持物化视图的数据库中，物化视图由数据库自动维护，无需触发器

  • 在数据复制到数据库的副本过程中，数据库系统会提供内在的数据库复制工具，无需使用

  • 数据库备份、拷贝过程中，禁止使用。应该在备份站点接管业务之后，再启动触发器。

    视图和游标

  • 视图：
    SQL允许通过查询来定义”虚关系”, 在概念上包含查询的结果, 虚关系并不预先计算并存储, 而是在使用需关系的时候才通过执行查询倍计算出来。不是逻辑模型的一部分，但作为虚关系对用户可见的关系称为：视图。
    数据库会存储视图的定义本身，而不会存储视图的查询表达式的执行结果。也就是说，一旦视图关系出现在查询中，它就会被已存储的查询表达式代替。（有点像内联函数那感觉）

    视图的优点:

    1. 便于用户使用简单的查询语句进行查询
    2. 维护数据的独立性,视图可以从多个表中检索数据
    3. 便于权限的管理
    4. 相同数据可以产生不同的视图。

    视图的缺点：

    1. 相对于物化视图而言，效率较低，是类似内联函数，在执行视图查询时，DBS会将视图转换成SQL再进行查询。
    2. 大部分情况不允许对视图进行修改，会引起不一致情况出现。

  • 游标：
    SQL会把SQL作用结果暂时保存在内存中，可以使用游标来显示、操作这个结果的部分，比如第一条、第二条，前十条等。一般不使用，用于逐条处理数据时会使用。

存储过程

预编译的SQL语句，优点: 允许模块化设计，只需一次创建，可以多次使用。并且会预编译，执行效率较高。缺点:移植性差

drop、truncate、delete

  • drop:DDL删除掉整个表，释放所有空间，不能回滚
  • truncate:DDL不会触发触发器，删除表中所有数据，不会为单独的删除进行日志记录，不能回滚
  • delete:DML会触发触发器，使用where可每次删除一行，同时会保存每个删除通过日志保存用以回滚

临时表

临时表只会在当前连接中可见，当连接关闭时，SQL会自动删除并释放所有空间，不同链接中可以创建同名表。

五大范式

  1. 1NF:原子属性不可再分

  2. 2NF:非主属性完全依赖于主码。消除了非主属性对码的部分依赖
    例如：关系表为（学号，姓名，课程名称，成绩，学分）
    其中（学号，课程名）为主码
    （学号，课程名）$\rarr$（姓名，年龄，成绩，学分）
    而由于存在
    （学号）$\rarr$（姓名，年龄）
    （课程名称）$\rarr$(学分)
    所以该关系表不符合第二范式

  3. 3NF:消除了非主属性对码的传递依赖
    例如：存在关系表（#学号，学院，学院主任）
    其中（#学号）$\stackrel{f}{\rarr}$（学院）
    又有（学院）$\stackrel{f}{\rarr}$（主任）
    即存在非主属性对主码的传递依赖，故不符合第三范式。

  4. BCNF:3NF基础上删除了主属性对码的传递依赖和部分依赖。
    换句话说符合BCNF的关系模式中：

    • 对于任何$\alpha\rarr\beta$或者$\alpha$是R的一个超码
    • 这是一个平凡函数依赖($\beta\subseteq\alpha$)

    例如：存在关系模式（学生，教师，课程）其中每个教师只教授一门课程，每门课程可以由多名教师教授。某个学生选定一门课便对应一个固定的教师。
    可以得到如下函数依赖：
    （学生，课程）$\rarr$（教师）；
    （学生，教师）$\rarr$（课程）；
    （教师）$\rarr$（课程）。
    其中（学生，课程）和（学生，教师）都是候选码。该模型符合3NF因为没有任何非主属性对候选码的传递依赖或部分依赖。但不属于BCNF因为教师是决定因素而教师不是超码。

  5. 4NF：消除了多值依赖

非关系型数据库和关系型数据库

非关系型数据库：

  • 性能：NoSQL基于键值对，无需SQL层的解析，性能较高。
  • 可扩展性：数据之间没有耦合性，容易扩展。

关系型数据库：

  • 复杂查询：可以用SQL语句方便的在一个表及多个表之间做复杂的数据查询。
  • 事务支持：使得对于安全性能很高的数据访问要求得以实现。

连接

  • 自然连接
  • 内连接：与自然连接类似，但不要求属性列同名，可以用using和on指定连接条件。并且不会删除连接条件那个列
  • 左外连接
  • 右外连接
  • 全外连接
  • 交叉连接：产生两个关系的笛卡尔积，不做选择。

char和varchar

  • char：
    • char长度固定，不可变会用空格填充，存取效率高，空间换时间
  • varchar：
    • varchar长度可变，时间换空间

DDL DML DCL

  • DDL:数据库定义语言
  • DML:数据库操作语言
  • DCL:数据库控制语言，可用于授权或收回权限，进行回滚，提交等操作。

为什么是3次握手而不是2次握手

三次握手具体：

  1. 客户端发给服务器SYN seq：确认服务器能建立连接
  2. 服务器发给客户端SYN ACK 并分配资源：确定服务器可以发送数据，可以建立连接，并且确认客户端的起始序列号
  3. 客户端第三次握手，并可以携带数据：确定客户端可以发送数据，并确认服务器端起始序列号

考虑这种情况，客户端发出SYN建立连接请求，之后该报文段超时，重发SYN建立连接，之后正常传输并关闭了连接。但是第一个超时报文段只是延时很久才到，并没有丢失，在关闭之后到达，这种情况下，服务器会进行连接的建立，但是客户端并没有数据要发送，导致了资源的浪费。

两次握手情况下，只有服务器对客户端的起始序列号做了确认，但客户端却没有对服务器的起始序列号做确认，不能保证传输的可靠性。

如何避免SYN DDos

  • 采用 SYN cookie 机制，服务器接收到SYN之后，并不知道是否是恶意攻击，因此不会打开半开连接，而是将一个秘密数，SYN源和目的IP地址与端口号进行散列函数，记录为cookie。服务器发送具有这种特殊初始序列号的SYNACK分组。服务器并不记忆该Cookie或任何对应于SYN的其他状态信息。
  • 如果用户合法，会返回一个ACK报文段。服务器接收到该segment，需要确认该ACK与之前的SYN是对应的。服务器会将之前发送的SYNACK segment中的源、目的IP地址和端口号以及秘密数进行相同的散列函数。如果该散列结果+1之后与客户的SYNACK确认值相同，则认为该连接合法。

为何四次挥手

因为建立连接时，服务器第二次握手只需回复SYN ACK而不需要携带实际内容。而在结束过程中，服务器收到客户端发来的结束segment之后，有可能还要发送许多消息，这些消息无法放在一个segment之中，所以需要四次挥手来保证服务器的所有消息传送完毕。

为何结束后提出结束方还需要等待两个MSL

MSL（segment最长生存时间 Maximum Segment Lifetime）：指segment在网络中最长生存时间，超出时间的segment会在接收后被丢弃。

  • 这可以保证，本次连接中的所有segment均已失活，不会出现本次连接中的segment出现在之后的网络环境中。
  • 试想这种情况：C发出了Fin segment，并且受到了对此确认，连接处于半关闭状态。之后S发出了所有segment之后也发送Fin segment，C收到之后对其作出ACK，并关闭了连接。但是最后一个ACK丢失了，此时S还处于未关闭状态，会不断在超时之后发送Fin segment，而C已经关闭了连接，不会再做出反应。导致S无法正常结束连接状态。

查询处理和查询优化

发表于 2020-04-20 更新于 2021-04-08

查询处理

查询处理是指从数据库中提取数据时涉及的一系列活动。这些活动包括：

  • 将高级数据库语言表示的查询语句翻译为能在文件系统的物理层上使用的表达式
  • 为优化查询而进行的各种转换，以及查询的实际执行。

概述

查询处理包括三个步骤：

  1. 语法分析与翻译
  2. 优化
  3. 执行

查询优化指：构造具有最小查询执行代价的查询执行计划，这是DBS的责任。

查询代价的度量

大型数据库中，最主要的代价是磁盘IO

表达式计算

物化

对关系代数表达式建立运算符树，之后自下而上开始运算，每次得到的结果都进行物化，建立起一个临时关系。这个临时关系还有可能需要写到磁盘中。

  • 某些情况下必须使用物化计算。
  • 物化计算：* 运算的没这种检结果被创建（物化），然后用于下一层的计算。

双缓冲： 使用两个缓冲区，一个用于连续执行算法，另一个用于写出结果。允许CPU和IO活动并行，从而提高算法执行速度。通过为输出缓冲区分配额外的磁盘块以及每次同时写出多个块，可以减少磁盘搜索速度。

流水线

通过减少查询中产生的临时文件数，可以提高查询效率。将多个关系操作组合成一个操作流水线实现，其中一个操作的结果传送到下一个操作，这样的计算称为流水线计算。

创建流水线的好处：

  1. 能够消除读和写临时关系的代价，减少查询计算代价。
  2. 如果一个查询计算计划的根操作符及其输入合并到流水线中，那么可以迅速开始产生查询结果。一旦他们生成，可以立即把结果显示给用户。

流水线可以按以下两种方式执行：

  1. 在一条需求驱动的流水线中，系统不停地向位于流水线顶端的操作发出需要元组的请求。每当一个操作收到需要元组的请求，它就计算下一个（若干个）元组并返回。该方式中，元组消极的按需产生。
  2. 生产者驱动流水线中，各操作并不等待元组请求，而是积极地产生元组。生产者驱动流水线中，各个操作作为系统中一个单独的进程或线程建模，以处理流水线输入的元组流，并产生相应的输出元组流。元组会放到输出缓冲区中，指导缓冲区已满，等待它的父操作将缓冲区中的元组移除。

关系代数运算的执行

选择运算

  • 线性搜索：慢，支持所有文件
  • 主索引，码属性等值比较
  • 主索引，非码属性等值比较
  • 辅助索引，等值比较

连接运算

  • 嵌套循环连接：代价最大，需要逐个检查两个关系中的每一对元组。
  • 块嵌套循环连接：内存缓冲区不能完全容纳一个关系式，使用块嵌套循环连接，以块的方式处理关系。在最坏情况下，对外层关系的每一个块读一次，而不是对外层关系的每个元组读一次。使用较小的关系作为外层关系更为有效，并且在内存中尽可能多的块用于存储外层关系的块
  • 索引嵌套循环连接：将元组数量较少的关系作为外层关系效果较好。
  • 归并连接：使用前最好对元组进行排序
  • 散列连接：用较少的关系作为构造用输入关系。

查询优化

查询优化：从许多策略中找出最有效的查询执行计划的一种处理过程。

查询执行计划的产生：

  1. 产生逻辑等价表达式
  2. 对所产生的表达式以不同的方式做注释，产生不同的查询计划
  3. 估计每个执行计划的代价，选择估计代价最小的一个

查询优化器中，上面3个步骤是交叉执行的：如产生一些表达式，加以注释后产生执行计划，之后再进一步产生一些表达式，再加以注释以此类推。

查询优化器使用启发式方法来减少优化代价：

  • 尽早执行选择运算
    例如：$\sigma_\theta(r\Join s)$其中$\theta$只引用$s$的属性。选择可以提前进行，然而如果$r$相对于$s$非常小，并且在$s$的连接属性上有索引，但在$\theta$属性的引用属性上无索引，那先做选择就需要在选择之后，对于$s$进行一次全体元组的扫描。这会加大代价。
  • 尽早执行投影运算（当投影涉及到了多个表中的属性时无法提前进行投影）

启发式算法并不验证采用本规则转换之后代价是否会减少，而是直接加以使用。

许多优化器允许为查询优化指定一个成本预算。当超过优化成本预算时停止搜索最有计划，返回当前的最优计划。

许多应用会反复执行同样的查询，不过查询中的常数不一样，例如经常会查询一个学生的所有信息，不同查询中只有学生ID不同，这种情况下，会将之前的最优计划进行缓存，缓存和重用查询计划成为计划缓存。

查询执行时间主要花费在慢速的磁盘IO上。

应用层

发表于 2020-04-20 更新于 2021-04-08

应用层

应用层选择传输层考虑的问题

  • 可靠数据传输：因为分组可能会在网络中失序，出错，丢失。有些应用不允许这种问题出现，所以需要提供可靠数据传输保证。一些不在意的程序，如：视频、音频，等能够承受小差错的称为容忍丢失的应用。
  • 吞吐量：运输层协议应该能够提供吞吐量保证，保证能够r比特/秒的确保吞吐量。例如因特网电话要对语言以32kbps的速率进行编码，如果运输层协议不能以该吞吐量传输，那么得到的那些几乎没有作用，因为无法在接收端复原，这类应用被称为宽带敏感应用，而不要求的如电子邮件、文件传输、HTTP等称为弹性应用。
  • 定时：运输层协议也能提供定时保证。这里指的是延迟，比如电话会议、视频、虚拟环境、游戏都要求数据交付有严格的时间限制。
  • 安全性：运输层协议能够为应用程序提供一种或多种安全性服务。

TCP UDP都没有提供任何的加密机制，这就是说发送进程传输到套接字的数据，与经网络传送到目的进程的数据相同。这可能会导致许多安全问题出现。

如今因特网研究了TCP的加强版本，称为安全套接字层（Secure Socket Layer, SSL）。用SSL加强后，不仅能做到TCP所能做到的一切，而且提供了关键的进程到进程的安全性服务，包括加密、数据完整性和端点鉴别。SSL的修改版本称为运输层安全性（Transport Layer Security, TLS）

  • SSL不是与TCP和UDP在同一层次上的第三种运输协议，而是TCP的加强，这种强化是在应用层上实现的。应用程序若想使用SSL的服务，则它需要在该应用程序的客户端和服务端包括SSL代码。SSL有他自己的套接字API，这类似于传统的TCP套接字API。当一个进程使用SSL时，发送进程向SSL套接字传递明文数据。

SSL在TCP和应用程序之间添加了一个SSL子层，通过SSL套接字与应用程序通信，通过TCP套接字与TCP通信。如果URL以https开始，则说明正在使用SSL。

HTTP

HTTP TCP 端口号 80

持续连接和非持续连接

因为HTTP使用TCP进行传输，所以必须决定，同一用户对同一服务器所发出的请求，是使用同一TCP连接进行数据传输，还是每个请求使用一条TCP连接。

  • 非持续连接：同一用户对同一服务器所发出的请求，每次请求使用一次TCP连接进行数据传输
  • 持续连接：同一用户对同一服务器所发出的请求，是使用同一TCP连接进行数据传输

HTTP默认是持续连接，流水线工作方式。

  • 流水线工作方式：不需要之前的发送确认返回，就可以进行下一次传输
  • 非流水线工作方式：必须在之前的发送返回确认之后，才可以开始下一次传输

信息熵

发表于 2020-04-19 更新于 2021-04-08

信息熵的概念

信息量的度量：香农使用“比特（Bit）”这个概念来度量信息量。例如，假设有32支球队，所有球队夺冠概率相等，那么“最终夺冠的球队是XXX”这条信息的信息量就是$log(32)=5$。但是，事实上32支球队的夺冠概率并不相等，我们已知某些队更强，并且他们夺冠的概率分别为 $p_1,p_2,\cdots,p_{32}$，那么，“最终夺冠的球队是XXX”所提供的信息量就是
$$
\begin{aligned}
H=-(p_1logp_1+p_2logp_2+\cdots +p_{32}logp_{32})
\end{aligned}
$$
香农将其称为“信息熵”，一般用符号$H$表示，单位是比特。

对于任意一个随机变量$X$，它的熵定义如下：
$$
\begin{aligned}
H(X)=-\sum_{x\in X}P(x)logP(x)
\end{aligned}
$$
变量的不确定性越大，熵也就越大。

条件熵

设$X$和$Y$是两个随机变量，$X$是我们需要了解的。假定我们现在知道了$X$的随机变量分布$P(X)$，那么也就知道了$X$的熵：
$$
\begin{aligned}
H(X)=-\sum_{x\in X}P(x)logP(x)
\end{aligned}
$$
那么$X$的不确定性就是这么大。现在假定我们还知道$Y$的一些情况，包括它和$X$一起出现的概率，在数学上称为联合概率分布。定义在$Y$的条件下的条件熵为：
$$
\begin{aligned}
H(X|Y)=-\sum_{x\in X,y\in Y}P(x,y)logP(x|y)
\end{aligned}
$$
可以证明$H(X)\geq H(X|Y)$，也就是说，知道了$Y$的信息之后，关于$X$的不确定性下降了。即二元模型的不确定性小于一元模型。

如果又有了另一个随机变量$Z$，那么可以定义有两个条件的条件熵：
$$
\begin{aligned}
H(X|Y,Z)=-\sum_{x\in X,y\in Y,z\in Z}P(x,y,z)logP(x|y,z)
\end{aligned}
$$
可以证明$H(X|Y)\geq H(X|Y,Z)$，也就是三元比二元更为确定。

下面问题就是：上面的等号何时会成立？

等号成立说明增加了信息，但不确定性没有降低，说明这两个变量之间没有任何关联。即，新增的变量并不能降低原有变量的不确定性。因此引出了互信息。

互信息

互信息（Mutual Information）是一个用于描述两个随机事件之间“相关性”的量化度量。

假定有两个随机事件$X$和$Y$，那么他们的互信息 $I(X;Y)$ 定义如下：

$$
\begin{aligned}
I(X;Y)=\sum_{x\in X,y\in Y}P(x,y)log\frac{P(x,y)}{P(x)P(y)}
\end{aligned}
$$

还有，$I(X;Y)=H(X)-H(X|Y)$。互信息的取值在$(0 , min\lbrace H(X),H(Y) \rbrace )$之间。当二者完全相关时，$I(X|Y)=H(X)$，同时$H(X)=H(Y)$；当二者完全无关时$I(X;Y)=0$

事务管理

发表于 2020-04-18 更新于 2021-04-08

事务管理

事务：构成单一逻辑工作单元的操作合称为事务，事务要么不发生，要么全部发生。即使有故障，数据库系统也必须保证事务的正确执行。同时，数据库应该引入一种能够避免会引发不一致性的方式来管理事务并发执行。

事务概念

事务是一个程序执行单元。事务通常由高级数据操纵语言或编程语言，如通过ODBC或JDBC嵌入式数据库访问来书写。begin transaction和end transaction来界定。事务由这两个定界符之间的所有操作组成。

  • 这些步骤集合必须作为一个单一的、不可分割的单元出现。此乃原子性（Atomicity）。

  • 事务是一个单一的、不可分割的单元出现，他不应该被不属于该事务的数据库操作影响。DBS必须采取特殊处理来保证事务能够正常执行，而不受到来自并发执行的数据库语句所干扰。此乃隔离性（Isolation）

  • 即使出现系统崩溃的情况出现，也要保证事务的操作是持久的。此乃持久性（Durability）

  • 事务必须保证数据库的一致性，即事务从一个一致性的数据库状态开始，事务提交之后数据库仍然是一个一致性状态。这种特性称为一致性（Consistency）

不一致状态：由于故障出现，系统不再反映世界的真实情况，称为不一致状态，不一致状态只能出现在事务的执行过程期间，在其他时间不可见。

持久性：
达到持久性的两种方法：

  • 在事务结束前，将更新写入磁盘
  • 在事务结束前，将已执行的更新信息已写入磁盘，该记录信息必须足够充分，能够保证出现故障之后事务可以重新执行，恢复该状态。

事务原子性和持久性

事务并非总能完成执行，这种未完成的事务称为中止（aborted）了。中止事务必须不对数据库的状态造成影响。一旦中止事务造成的变更被撤销，我们说事务已回滚（rolled back）。

恢复机制负责管理事务中止。常见的方法是维护一个日志（log）。

成功完成的事务称为已提交（committed）。一个已提交的事务会使数据库进入一个新的状态，即使出现故障，这个状态也必须保持。

一旦事务已提交，我们不能通过中止来撤销其造成的影响。撤销已提交事务的唯一方法是执行一个补偿事务，这由用户负责，不由DBS处理。

事务进入中止状态后，有两种选择：

  • 重启事务，仅当引起事务中止的是硬件导致的外中断。重启事务是一个新的事务。
  • 杀死事务，当事务内部逻辑异常或输入异常时，则需要杀死事务。

事务隔离性

允许事务并行好处：

  • 提高吞吐量和资源利用率
  • 减少等待时间

在并发执行中，必须保证执行的任何调度的效果都和没有并发执行调度的效果一样，这样可以保证数据库的一致性。也就是说，调度应该在某种意义上等价于一个串行调度。这种调度称为可串行化调度

类似于OS中的并发控制。

可串行化

如果调度$S$可以经过一系列非冲突指令交换成$S’$，我们称$S$和$S’$是冲突等价的。

可恢复性

不论什么原因，如果事务$T_i$失败了，我们必须撤销该事务的影响以确保其原子性。在允许并发的系统中，原子性要求依赖于$T_i$的事务$T_j$(即读取了$T_i$写的数据)也中止。为确保这一点，我们需要对系统允许的调度类型做一些限制。

可恢复调度

一个可恢复调度应满足：依赖者在被依赖者提交之后再提交。

无级联调度

通过对调度加以限制，依赖者只能在被依赖事务提交之后再进行读。

事务并发会出现的问题

  • 脏读：读取了尚未提交的事务所更新的数据。例如：事务A读取了事务B更新的数据，然后B回滚操作，那么A读取到的数据是脏数据。
  • 不可重复度：事务 A 多次读取同一数据，事务 B 在事务A多次读取的过程中，对数据作了更新并提交，导致事务A多次读取同一数据时，结果因此本事务先后两次读到的数据结果会不一致。
  • 幻读：幻读解决了不重复读，保证了同一个事务里，查询的结果都是事务开始时的状态（一致性）。
    例如：事务T1对一个表中所有的行的某个数据项做了从“1”修改为“2”的操作 这时事务T2又对这个表中插入了一行数据项，而这个数据项的数值还是为“1”并且提交给数据库。 而操作事务T1的用户如果再查看刚刚修改的数据，会发现还有跟没有修改一样，其实这行是从事务T2中添加的，就好像产生幻觉一样，这就是发生了幻读。

    事务隔离性的级别

如果事务在独立执行时保证数据库一致性，那么可串行性就能保证事务在并发执行时也具有一致性。

SQL也允许一个事务这样规定：它以一种与其他事务以不可串行化的方式执行。

SQL标准隔离型级别如下（一致性级别由高到低）：（重点）

  • 可串行化：保证可串行化调度，一致性最高，就好像没有并行。
  • 可重复读：只允许读取已提交事务，而且在一个事务的两次读取同一个数据项期间，其他事务不可更新该数据。但不要求与其他事务可串行化。
  • 不可重复读（已提交读）：只允许读取已提交数据，但不要求可重复读。例如，在事务两次读取一个数据项期间，另一个事务更新了该数据并提交。
  • 未提交读：允许读取未提交数据。

以上所有隔离性级别都不允许脏写，即如果一个数据项已经被另一个尚未提交或是中止的事务写入，则不允许对该数据项进行写操作。

并发控制

使用三种方式实现并发控制：

  1. 加锁
  2. 时间戳
  3. 有效性检查

其中2，3两种方法是基于恢复系统的控制方法。最常用的是两阶段锁和快照。

基于锁的协议

锁

  • 共享锁：$T_i$获得了$Q$上的共享锁，则可读不可写
  • 排他锁：$T_i$获得了$Q$上的排他锁，则可读可写

锁必须由事务提出申请，并得到了并发控制器的授予，之后才能访问数据。并且，在数据访问期间必须拥有相应的锁，同时，即使在最后一次访问数据后立刻释放锁也不一定是正确的。

然而，这种封锁可能会导致死锁出现。我们要求系统中每个事务遵循封锁协议。

锁的授予

申请排他锁的事务可能由于不断出现的申请共享锁的事务出现而饿死。为了避免这种情况，我们可以考虑读写者问题中，写者优先的方式，给申请锁的操作加锁。

两阶段锁！

保证可串行的一个协议是：两阶段封锁协议，该协议要求事务分为两个阶段，提出加锁和解锁申请。

  1. 增长阶段：该阶段可以申请锁，但是不能释放锁
  2. 缩减阶段：该阶段可以释放锁，不能申请锁

二者的分界点（最后一次加锁的位置）称为事务的封锁点。

  • 两阶段锁不能保证不会发生死锁。不能避免级联回滚的发生。

为了避免级联回滚，可以采用严格两阶段封锁协议，该协议除了要求两阶段之外，还要求实物持有的所有排他锁必须在事务提交之后才可以释放。

强两阶段锁：要求事务提交之前不得释放任何锁。

多粒度：

某些情况允许我们把多个数据项聚成一组，作为一个同步单元。

这就需要系统定义一种多级粒度机制。通过各种大小的数据项并定义数据粒度的层次结构。小粒度嵌套在大粒度的数据项中来实现。可以形象的表示为树。

树中的每个节点都可以单独显式的加锁。当一个节点加锁时，该事务也以同样的锁，隐式的封锁其所有后代节点。

这种条件下，系统在一事务申请锁的时候，需要搜索整棵树判断是否可以授予锁，这破坏了多粒度的初衷。

为解决这种情况，引入了一种意向锁。如果给一个节点加上了意向锁，则意味着树较低层也需要显示加锁。在一个节点显式枷锁之前，该节点的全部祖先均加上了意向锁。
希望给某个节点（$Q$）加锁的事务必须遍历从根到$Q$的路径。遍历过程中，给各节点加上意向锁。

时间戳

对系统中每个事务，我们使用唯一固定的时间戳与其联系起来，时间戳记为$TS(T_i)$，时间戳是事务开始执行前DBS赋予的。后来的新事务的时间戳大于先前的事务。时间戳可有两种方法实现：

  • 系统时钟
  • 逻辑计数器

每个数据项需要与两个时间戳值相关联：

  • W-timestamp(Q) ：表示成功执行write(Q) 的所有事务的最大时间戳
  • R-timestamp(Q) ：表示成功执行read(Q) 的所有事务的最大时间戳

时间戳排序协议：一种保证任何有冲突的read或write操作按时间戳顺序执行，保证无死锁，但可能会导致长事务饿死。

  1. 假设事务$T_i$发出read(Q):
    1. $TS(T_i)<$W-timestamp(Q)，意味着这个事务开始的早，还没读呢就有人又给写了点东西覆盖了Q，则$T_i$读入值会被覆盖，回滚。
    2. $TS(T_i)\geq$W-timestamp(Q)，成功。R-timestamp被置为$TS(T_i)$和R-timestamp之中最大值。
  2. 假设事务$T_i$发出write(Q):
    1. $TS(T_i)<$R-timestamp，意味着这个写事务开始的早，但是还没写完呢，就有人读了。所以$T_i$产生值Q是先前需要值，回滚。
    2. $TS(T_i)<$W-timestamp，意味着这个事务开始的早，还没写完提交呢，又有人提交了个写，$T_i$写入值Q已过时，回滚。
    3. 其他情况，设置W-timestamp等于$TS(T_i)$

基于有效性检查

有效性检查协议要求每个事务$T_i$在其生命周期中按两个或三个阶段执行，这取决于该事务是一个只读事务还是一个更新事务。

  1. 读阶段：在这一阶段，数据项会被读入并保存在$T_i$的局部变量之中，不会对数据库进行真正的更新。
  2. 有效性检查阶段：对事务$T_i$进行有效性测试。判定是否可以执行write操作而不违反可串行性。如果事务有效性测试失败，系统会终止这个事务。
  3. 写阶段：如果通过了有效性检查，会将局部变量更新到数据库中。（只读无此阶段）

调度器会维护三个事务集合：

  1. START集合：已开始但尚未有效性确认的事务集合
  2. VAL集合：已确认但未写的事务集合
  3. FIN集合：已完成第三个阶段写的事物的集合

事务$T_i$的有效性测试要求任何满足$TS(T_k)<TS(T_i)$的事务$T_k$必须满足下面两个条件之一：

  1. Finish（$T_k$）< Start($T_i$)。即$T_k$在$T_i$开始之前完成。
  2. $T_k$所写的数据与$T_i$所读的数据不相交，并且$T_k$的写阶段在$T_i$开始有效性检查之前完成。

有效性检查机制可以自动预防级联回滚，但有可能长事务会饿死。

有效性检查机制事务乐观的执行，假定能够完成并且最终有效，因此也称乐观并发控制，而时间戳和加锁是悲观的，检测到冲突时，强迫等待或是回滚。

恢复系统

我们把存储系统分为内存和非易失性存储区。数据库的大部分内容存在于非易失性存储器中，在特定时刻，只有少量的数据存在于主存中。

每个事务$T_i$都有他自己的私有工作区，用于保存$T_i$所访问及更新的所有数据项的拷贝。该工作区在$T_i$初始化时由系统创建；在$T_i$提交或中止时由系统删除。

在事务提交之后，它所执行的更新并不一定会马上存储在非易失性存储器中，而是采取延迟写的方法，进行批量写入。所以在这期间，有可能出现系统崩溃等情况，导致更新提交之后却未写入非易失性存储器。

日志

为了避免这种情况，多数的数据库系统采用了日志方式，用于保证事务的原子性，要么全部发生，要么不发生。

一条日志<$T_i, X_j, V_1, V_2$> 包括了以下几个部分：

  1. 事务标识：是执行事务操作的唯一标识。
  2. 数据项标识：指明了所操作的数据项。
  3. 旧值：写前值。
  4. 新值：写后值。

恢复算法

恢复算法需要考虑两种情况：

  • 提交后修改没能更新到非易失性存储器中。
  • 修改已经更新到非易失性存储器了，但由于故障，这个事务需要中止。

恢复算法需要防止级联情况的出现，所以需要使用严格两阶段锁来实现。

一个事务的commit日志记录输出到稳定存储器后，我们就说这个事务提交了。

恢复系统应该包括两种功能：

  • undo($T_i$)撤销
  • redo($T_i$)重做

这两种功能需要系统从尾端反向扫描日志来执行回滚。

  • 如果日志包括<$T_i$start>，但既不包括<$T_i$commit>也不包括<$T_i$abort>，则需要对事务$T_i$进行撤销。
  • 如果日志包括<$T_i$start>，以及<$T_i$commit>或<$T_i$abort>，则需要对事务$T_i$进行重做。事务即使abort了也需要重做是因为，日志中包括了原本abort之后的操作，而我们又是顺序执行，所以简单重放即可（会包含一定冗余，但简化了形式）。

检查点

如果系统故障，我们需要从头进行整个日志，则需要耗费大量的时间。所以引入了检查点概念。

检查点：

  1. 在执行检查点操作过程中，不允许任何更新
  2. 在执行检查点的过程中，将所有更新过的缓冲块都输出到磁盘。

检查点的执行过程如下：

  1. 将当前主存中的日志记录输出到稳定存储器
  2. 将所有修改的缓冲块输出到磁盘。
  3. 将一个日志记录$\text{}$输出到稳定存储器。

之后，如果系统崩溃了，只需要反向搜索找到最后一个$\text{}$，并对在检查点之前开始，但未<commit>或是<abort>的事务（这些事务还未完成，不会更新到持久性设备），或是检查点写到日志之后开始的事务，执行以下操作：

  • 并且在日志之中没有执行<commit>或是<abort>的事务进行撤销
  • 对在检查点写到日志之后开始的事务，并且在日志之中有执行<commit>或是<abort>的事务进行重做

OSI七层模型

发表于 2020-04-18 更新于 2021-04-08

应用层

最复杂的一层，用户与网络的接口

常见服务：

  • FTP
  • SMTP
  • HTTP

表示层

用于处理在两个信道系统中交换信息的表示方式（语法和语义）

功能：

  • 数据格式变换
  • 数据的加密和解密
  • 数据的压缩和恢复

主要协议：HTML、ASCII

会话层

向表示层实体/用户进程提供建立连接，并在连接上有序的传输数据。用于建立同步（SYN）。

功能：

  • 建立、管理、终止会话
  • 建立校验点，通信失效时，从校验点恢复通信。（如传输大文件）

主要协议： 略

传输层

负责主机的两个进程间的通信，即端到端的通信。传输单位是报文段或用户数据包。

主要功能：

  • 可靠传输、不可靠传输
  • 差错控制：报文丢失、错误，报文顺序错误
  • 流量控制：端到端之间的传输流量，控制发送方的发送速率。
  • 复用：多个应用层同时使用下层运输层的服务，通过端口号进行区别。
  • 分用：通过不同的端口号进行识别，分别交付，不同的端口号之间互不干扰。

主要协议：TCP、UDP

网络层（IP层）

把分组从源端传送到目的端，为分组交换网上的不同主机提供通信服务。网际互连

网络层的传输单位：数据报

数据报过长时，会将数据报分割成小的分组（packet）。

功能：

  • 路由选择：选择路由器之间的传输路径
  • 流量控制：限制发送方的发送速度
  • 差错控制：保证传输层数据无错，出错分组丢弃。
  • 拥塞控制：宏观控制，控制全局。如果所有节点都来不及接收分组，将要丢失大量分组时，网络处于拥塞状态，需要采取一定措施来缓解。

数据链路层

把网络层传下来的数据报组装成帧（注：只有链路层会加头加尾）

链路层的传输单位是帧

功能：

  • 成帧（定义帧的开始和结束）
  • 差错控制：帧错、位错
  • 流量控制：控制发送方发送速率
  • 信道访问控制

常见协议：HDLC PPP SDLC STP

物理层

将比特流转换成电信号，在物理媒体实现比特流的透明传输

物理层传输单位是比特

功能：

  • 定义接口特性：引脚数目、插头
  • 定义传输模式：单工通信、半双工通信、全双工通信
  • 定义传输速率
  • 比特同步：发送方发送比特之后，接收方能够根据一定时序接收比特。
  • 比特编码：曼彻斯特、差分曼彻斯特编码

TCP/IP参考模型

发表于 2020-04-18 更新于 2021-04-08

先产生了协议栈，之后根据协议栈进行划分，分为4层

应用层

包含了OSI中的上三层（应用层、表示层、会话层）

对应协议：HTTP FTP DNS

传输层

传输层

对应协议: TCP UDP

网际层

IP | ICMP | OSPF

网络接口层

指出数据需要通过一定方式连接到物理介质上。

常见协议：Ethernet | ATM | Frame Relay

TCP/IP和OSI参考模型的异同

相同点：

  • 都采取了分层模型
  • 基于独立的协议栈概念
  • 可以实现异构网络互联

不同点：

  • OSI定义了三点：服务、协议、接口
  • OSI先出现分层，不指定特定协议
  • TCP/IP设计之初考虑到异构网络问题，将IP层作为重要层次
ISO/OSI模型 TCP/IP模型
传输层 面向连接 无连接+面向连接
网络层 无连接+面向连接 无连接

TCP/IP中注重IP层，认为面向连接是上层的事情，不需要IP层考虑，所以网络层只有无连接类型。

计算机网络概述

发表于 2020-04-18 更新于 2021-04-08

概念、组成、功能、分类

计算机网络：一个将分散的、具有独立功能的计算机系统。通过通信设备和线路连接起来，由功能完善的软件实现资源共享和信息传递的功能。
计算机网络是互联、自治的集合。

功能：

  • 数据通信
  • 资源共享（软件、硬件、数据）
  • 分布式系统
  • 提高可靠性
  • 负载均衡

组成：

  • 硬件
  • 软件
  • 协议：规定了数据传输的规范

工作方式：

  • 边缘部分：用户直接使用
  • 核心部分：边缘部分服务

功能组成：

  • 通信子网：实现数据通信。是各种通信介质、通信设备、相应的网络协议组成。
  • 资源子网：实现资源共享/数据处理。是实现资源共享功能的设备和软件的集合

资源子网：应用层 表示层 会话层
通信子网：网络层 链路层 物理层

计算机网络分类

  1. 按分布范围分：

    • 广域网（交换技术）
    • 城域网
    • 局域网（广播技术）
    • 个人区域网

  2. 按使用者分：

    • 公用网络
    • 专用网络

  3. 按交换技术分：

    • 电路交换
    • 报文交换
    • 分组交换

  4. 按拓扑结构分：

    • 总线型
    • 星型
    • 环形
    • 网状型

  5. 按传输技术分：

    • 广播式网络：共享公共通信信道
    • 点对点网络：分组存储转发、路由选择机制

标准化工作

标准分类：

  • 法定标准：OSI
  • 事实标准：TCP/IP

制定标准流程：RFC（Request For Comments）

  • 因特网草案：此阶段还不是RFC文档
  • 建议标准：开始成为RFC文档
  • 草案标准：已取消该阶段
  • 因特网标准

国际标准化组织ISO： OSI参考模型、HDLC协议
国际电信联盟ITU：制定电气通信规则
IEEE：学术机构、IEEE802标准
IETF：RFC

速率、带宽、吞吐量

速率

速率：数据率或称数据传输率或比特率。
连接在计算机网络上的主机在数字信道上传送数据的位数的速率。
单位：b/s kb/s Mb/s Gb/s Tb/s $10^3$进制
1 Byte = 8 bit

带宽

带宽原指某个信号具有的频带宽度，即最高频率与最低频率之差，单位是赫兹（Hz）。
计算机网络中，带宽表示网络的通信线路传送数据的能力，通常是单位时间内从网络中某一点到另一点所能通过的最高数据率，单位是“比特每秒”
