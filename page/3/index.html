关系数据库

发表于 2020-04-08 更新于 2021-04-08

关系数据库的结构

关系数据库由表的集合构成，每个表有唯一的名字。
一般来说，表中的一行代表了一组值之间的一种联系。
关系指代表，而元组指代行，属性指代列。
关系实例表示一个关系的特定实例。
对于关系的每个属性，都存在一个允许取值的集合，成为该属性的域。如果域中的元素被看作是不可再分的单元，则域是原子的。
空值（null）是一个特殊的值，表示值未知或是不存在。

数据库模式

关系模式由属性序列及各属性对应域组成。
例如: 一个department关系,它的关系模式是department(dept_name, building, budget)

码

  • 超码是一个或多个属性的集合, 这些属性的组合可以使我们在一个关系中唯一的标识一个元组。
  • 候选码：超码之中，任何真子集都不能成为超码的最小超码，称作候选码。
  • 主码：被数据库设计者选中的，用来在一个关系中区分不同元组的候选码。主码应该选择那些从不或很少发生变化的属性。一般会在关系模式中的主码属性加上下划线。
  • 外码：关系模式r1的属性中包括另一个关系模式r2的主码, 这个属性在r1上被称作参照r2的外码。关系r1也称为外码依赖的参照关系，r2叫做外码的被参照关系。
  • 参照完整性约束：参照关系中任意元组在特定属性上的取值等于被参照关系中某个元组在特定属性上的取值。

关系查询语言

  • 关系代数是过程化的，即用户指导具体过程。
  • 元组关系演算和域关系演算是非过程化的。

SQL数据定义

数据库中的关系集合必须由DDL指定给系统. DDL定义一组关系,同时还能定义每个关系的信息, 包括:

  • 每个关系的模式
  • 每个属性的取值类型
  • 完整性约束
  • 每个关系维护的索引集合
  • 每个关系的安全性和权限信息
  • 每个关系在瓷盘上的物理存储结构

例:
create table department
(
dept_name varchar(20),
building varchar(15),
budget numeric(12, 2),
primary key (dept_name));

create table course
(
course_id varchar(7),
title varchar(50),
dept_name varchar(20),
credits numeric(2, 0),
primary key (course_id),
foreign key (dept_name) references department );

insert into instructor values (20211, ‘Smith’, ‘Biology’, 66000);

delete from student; //会删除student中的所有元组,但是会保留关系student

drop table student; //会删除掉关系student,在没有create之前无法再次添加元组

alter table r add A D; //在r中添加域为D的属性A

alter table r drop A; //删除r中的属性A

SQL查询基本结构

单关系查询

select (distinct/all) name from instructor;

select name
from instructor
where dept_name = ‘Comp.Sci’ and salary > 70000 ;

多关系查询

select name, instructor.dept_name, building
from instructor, department
where instrctor.dept_name = department.dept_name ;
其中from会生成几个关系的笛卡尔乘积,之后通过where所对应的谓词选择相应的元组，最后select选择出所需的属性。

自然连接

例：select name, course_id from instructor natural join teaches;

生成的结果首先会列出二者的共同属性,之后是只出现在第一个关系中的属性,最后是只出现在第二个关系中的属性。

更为一般的
select A, B, C, D…
from r1 natural join r2 natural oin r3 …. natural join rm
where P;

同时, 为了发扬自然连接的优势, 并且能够避免不必要的相等属性所做出的限制, SQL提供了允许用户制定哪些列相等自然连接构造方式

select name, title
from (instructor natural join teaches ) join course using (course_id) ;
//instructor和teaches自然连接,之后与course建立连接,不过只要求course_id取值相同

附加操作

更名运算

select T.name, S.course_id
from instructor as T, teaches as S
where T.ID = S.ID ;

重命名关系的另一个作用是为了适用于需要比较同一个关系中的元组的情况。例如，找出满足下面条件的所有教师的姓名，他们的工资至少比Biology系某一个教师的工资要高。
select distinct T.name
from instructor as T, instructor as S
where T.salary > S.salary and S.dept_name = ‘Biology’;
不能简单地使用instructor因为无法表明是拿一个instructor。

字符串操作

  • %可以适配任意字符串
  • _可以适配任意一个字符
  • 通过‘\’作为转义字符, 可以表达原本的含义, 如 ‘\%’ ‘\_’ ‘\\‘

例:
select dept_name
from department
where building like ‘%Watson%’ ;
可以选择出所有所在大楼名字中包含Waston的系名字

排列元组的显示次序

select name
from instructor
where dept_name = ‘Physics’
order by salary desc, name asc;
desc 标识降序排列 asc 表示升序排列

空值

涉及空值的任何运算, 比较的结果都是unknown(包括null = null)
SQL允许使用is unknown / is not unknown来测试表达式的结果是否为unknown

  • and: true and unknown == unknown,
    false and unknown == false,
    unknown and unknown == unknown
  • or: true or unknown == true,
    false or unknown == unknown,
    unknown or unknown == unknown
  • not: not unknown == unknown

{(‘A’, null) , (‘A’, null)} 中的两个(‘A’, null) 会被视为相同的元组,如果使用distinct 只会保留一个元组， 而all会将二者保留

聚集函数(group by)

  • min
  • max
  • count
  • avg
  • sum

having子句会在聚集函数之后进行判断

数据库系统引言

发表于 2020-04-06 更新于 2021-04-08

引言

数据库系统（DBS）是一个相互关联的数据和一组用以访问这些数据的程序集合。

数据的集合成为数据库（DB）

数据库的目标

在数据库系统出现之前,通常由文件处理系统来组织信息。文件处理系统存在着以下问题：

  • 数据的冗余和不一致：相同信息重复存储；同一数据和其副本不一致
  • 数据访问困难：检索困难
  • 数据孤立：数据分散在不同的、格式不同的文件中
  • 完整性问题：DBS必须满足一些一致性约束。
  • 原子性问题：难以保证原子性
  • 并发访问控制：并发的更新操作导致数据的不一致
  • 安全性问题：权限管理困难

数据视图

数据抽象

开发人员通过以下几个层次的抽象，来对用户屏蔽复杂性，用以简化系统和用户的交互：

  • 物理层：底层，数据存储方式
  • 逻辑层：数据库中存储什么数据以及这些数据之间的关系。通过少量相对简单的结构描述了整个数据库。数据库管理员使用抽象的逻辑层，确定数据库中保存哪些信息。
  • 视图层：最高层次的抽象，方便用户和系统交互。

实例和模式

特定时刻存储在数据库中信息的集合称作数据库的一个实例。数据库的总体设计称作数据库模式

  • 物理模式：在物理层描述数据库的设计
  • 逻辑模式：在逻辑层描述数据库的设计
  • 数据库在视图层可以有几种模式，称作子模式，描述了数据库的不同视图。

程序员使用逻辑模式来构造数据库应用程序，物理模式隐藏在逻辑模式下，与具体程序相互独立。如果应用程序不依赖物理模式，则称为物理数据独立性。

数据模型

数据库结构的基础是数据模型数据模型是一个描述数据、数据联系、数据语义以及一致性约束的概念的工具的集合。
数据模型：

  • 关系模型：用表的集合来表述数据和数据间的联系。
  • 实体关系模型：ER模型，现实世界由一组称作实体的基本对象和这些对象间的联系构成。
  • 基于对象的数据模型：可看作是ER模型增加了封装、方法（函数）和对象表示等概念后的扩展。
  • 半结构化数据模型：允许相同类型的数据项，含有不同属性集的数据定义。XML被广泛的用来表示半结构化的数据。

数据库语言

数据操纵语言（Data Manipulation Language，DML）

用于增删改查数据的一种语言。
通常有以下两种类型：

  • 过程化DML：要求用户指定所需数据以及如何获得这些数据
  • 声明式DML：用户只需指定需求什么数据，不指明如何获得这些数据。

数据定义语言（Data Definition Language， DDL）

数据库模式的定义通过DDL来表达。数据库的存储结构和访问方式通过DDL来说明。
存储在数据库中的数据必须满足一些一致性约束：

  • 域约束：声明一种属性属于某种具体的域，数据必须符合它的域（如：整数、字符、日期）。
  • 参照完整性：一个关系中给定属性集上的取值在另一属性集中出现。
  • 断言：数据库需要时刻满足的某一条件。（例如，一个学生一学期只能选5门课），断言创建以后，系统会检测断言有效性，只有不破坏断言的数据库更新才会被允许。
  • 授权：不同的用户给予不同的权限，如读权限，插入权限（不可修改），更新权限，删除权限。

DDL以一些指令作为输入，产生一些元数据作为输出。元数据是数据的数据，包含在数据字典中。在用户读和修改数据之前，会先参考数据字典。
