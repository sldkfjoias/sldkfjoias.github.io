26尽可能延后变量定义出现的时间

  • 应该尽量将变量的定义延后，这样可以避免之前抛出异常，导致引起变量没有用到，却引发了构造和析构的开销。
  • 应该尽可能延后到定义变量时能够直接给变量初试实参，这样可以避免构造非必要对象以及无意义的default构造。

分析：

  • 做法A：1个构造函数 + 1个析构函数+ n个赋值操作
  • 做法B：n个构造函数 + n个析构函数

当class的赋值成本低于一组构造+析构的成本时，做法A相对高效，但做法A导致名称w的作用域比做法B更大。 所以除非：1.明确赋值成本低于“构造 + 析构”成本 2.代码对效率的敏感程度非常高。否则还是B更好

27尽量少做类型转换（待补充）

四种类型转换：

  • const_cast:通常用来移除const，是c++唯一有此能力的类型转换操作符
  • dynamic_cast:主要用来“安全项下转型”，唯一无法有旧式语法执行的动作。会消耗大量的成本
  • reinterpret_cast:不可移植，是一种低级转型
  • static_cast:隐式转换，最常用的应该就是这个，non-const -> const ,void* -> typed*, int -> double

28避免返回handles指向对象内部成分

避免返回指向对象内部的引用、指针、迭代器。遵守这个条款可以帮助增加封装性，帮助const成员函数像个const，并将空悬指针的可能性降到最低。

  1. 成员变量的封装性最多只等于“返回其reference的函数的访问级别” 即上述虽然ulhc和lrhc被声明为private，但public函数中upperLeft和lowerRight传出了它们的引用，所以变为了public封装性
  2. 如果const成员函数传出一个reference，后者所指数据与对象自身有关联，但存储在对象之外(ulhc和lrhc不存储在Rectangle中)，那么这个函数调用者可以修改那个数据。

29异常安全

异常安全函数提供下述三个保证之一：

  1. 基本承诺：如果异常抛出，程序内任何事物仍保持在有效状态下。没有任何对象或数据结构会因此而败坏，所有对象都处于一种内部前后一致的状态。但程序的现存状态并不能预测。即1->2的过程中抛出异常，有可能返回状态1，也有可能到达一个默认状态。
  2. 强烈保证：原子性，如果异常被抛出，程序状态不改变。调用这样的函数如果成功就是完全成功，如果失败就会恢复到调用之前的状态。
  3. 不抛掷(nothrow)保证：承诺绝不抛出异常！作用于内置类型上的所有操作都提供nothrow保证。

32public继承意味着is-a关系

如果class D以public关系继承class B，便是告诉编译器，每一个类型为D的对象同时也是一个类型为B的对象，反之不成立。也就是里氏替换原则，热河一个B可以使用的地方，都可以使用D替换，反之不成立。private和protect则完全不同

34区分接口继承与实现继承

  • 成员函数的借口总是会被继承。public继承意味着is-a，所以对base class为真的任何事情对于derived class一定为真。

对于pure virtual函数，有两个特性：

  1. 它们必须被任何“继承了它们”的具象class重新生命
  2. pure virtual函数在抽象基类中没有定义
  • 声明一个纯虚函数的目的是为了让derived class只继承函数接口

但是可以在抽象基类中为纯虚函数提供定义，其唯一调用方法是ptr2Derived->Base::pureVirtualfunc();

  • 声明普通虚函数的目的：让derived class继承该函数的接口和缺省实现

但是继承缺省实现可能带来的问题：添加了一个新的派生类时，忘记实现该派生类的函数实现，从而继承了错误的缺省函数。解决方法：切断“virtual函数接口”和“缺省实现”之间的联系，除非派生类明确需要这个缺省函数。

  1. 将所有成员函数声明为non-virtual，这导致派生类不够灵活，无法进行特化工作。当然这对于一个不想成为base-class的类来说没什么问题
  2. 将所有成员函数声明为virtual，这只对Interface class这些类是正确的。因为某些函数不应该被派生类重新定义。

35考虑virtual函数以外的其他选择

可以考虑以下方式替代virtual函数：

  • 使用non-virtual interface手法，它以public non-virtual成员函数保过访问性较低的（protected、private）的virtual函数。Template Method设计模式的一种特殊形式
  • 将virtual函数替换为“函数指针成员变量”，Strategy设计模式的一种分解表现形式
  • 以tr1::function成员变量替换virtual函数，因而允许任何可调用物（callable entity）搭配一个兼容于需求的签名式。Strategy设计模式的某种形式
  • 将继承体系内virtual函数替换成另一个继承体系内的virtual函数。这是Strategy设计模式的传统实现手法

36绝不重新定义继承而来的non-virtual函数

如果mf是个non-virtual函数，同时D拥有自己版本的mf函数，那么由于non-virtual是非虚函数，所以其都是静态绑定，也就造成了二者不同

另一方面，如果mf()是一个virtual函数，那么不论通过pB还是pD调用mf，都会调用D::mf()，因为pB和pD真正指向的都是一个类型为D的对象。

简而言之：这一条意思就是，不重新定义继承而来的non-virtual函数，如果需要D重写，那你B就该把那个函数定义成一个虚函数。要不然你就别public继承，因为D重写就违反了is-a法则。

由此，又可以印证#7中的base class应该定义成virtual函数，因为derived class不应该重新定义继承来的non-virtual函数

38 has-a关系

  • 复合的含义与public继承完全不同
  • 在应用域，复合意味着has-a；在实现域，复合意味着is-implemented-in-terms-of（以xx实现出）

针对第二点举个例子：比如要实现一个自己的set集合类，为了节省空间考虑通过list实现，显然不能用public继承，因为public继承意味着is-a，但是set并不包含重复元素，list是可包含的，所以说set is-a list是错误的，而应该是set包含一个list的成员。这就是is-implemented-in-terms-of。

39谨慎使用private继承

  • Private继承意味着is-implemented-in-terms-of。它通常比复合的级别低。但是当derived class需要访问protected base class成员时，或者需要重新定义继承而来的virtual函数时，这么设计是合理的。
  • 与复合不同，private继承可以造成empty base最优化。这对于“将对象空间最小化”的开发者来说可能很重要。

40多重继承和虚继承

关于virtual base class的忠告：

  • 非必要不使用virtual bases，平常就是用non-virtual继承。
    • 因为虚继承产生的对象会比non-virtual继承的对象体积大，访问速度变慢。这就代表着很多额外的开销，包括时间和空间；
    • 同时virtual base classes初始化的规则更加复杂且不直观：virtual base的初始化责任是由继承体系中的最低层class负责，也就暗示：（1）classes若派生字virtual bases而需要初始化，必须认知其virtual bases（2）当一个新的derived class加入继承体系内，它必须承担其virtual bases的初始化责任。
  • 如果必须使用virtual base classes，那么尽可能避免在其中放置数据，这样可以避免诡异的初始化和赋值操作。

41了解隐式接口和编译器多态

  • classes和templates都支持接口和多态
  • 对class而言接口是显式的，以函数签名为中心。多态则是通过virtual函数发生于运行期
  • 对template参数而言，接口是隐式的，奠基于有效表达式。多态则是通过template具现化和函数重载解析发生于编译器
