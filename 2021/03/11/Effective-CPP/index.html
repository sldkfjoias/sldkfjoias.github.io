2.尽量以const，enum，inline替换#define

使用const double AspectRatio = 1.653替代#define ASPECT_RATIO 1.653，因为后者在调试时不会显示ASPECT_RATIO而是显示1.653从而给调试带来困难。同时define也会导致object code中出现大量重复的1.653，从而导致更大的空间占用。

  2. 方便自己看懂别人

使用template inline 替代 #define函数

类似#define CALL_WITH_MAX(a, b) f((a) > (b) ? (a) : (b))的宏看起来像函数，但是不会引起函数调用的开销。这个函数夹杂着宏实参，使用较大的参数调用函数f。
对于这种宏需要对宏中所有实参加上小括号，否则会出现问题，同时即使所有实参都加上小括号，依然会引发奇怪的行为。

对于这种情况，可以使用template inline函数带来宏的效率以及一般函数的所有可预料行为和类型安全性。

总结

  • 对于单纯常量，最好以const对象或enum替换#define
  • 对于形似函数的宏，改用inline函数替代#define

non-local static对象初始化

对于non-local static对象：由于本编译单元可能使用其他编译单元的 non-local static 变量进行初始化，而存在无法保证其初始化的相对顺序，所以需要特定的解决方法：将每个non-local static对象搬到自己的专属函数内（该对象在此函数内被声明为static）。这些函数返回一个reference指向它所含的对象。然后用户调用这些函数，而不直接指涉这些对象。这样，non-local static对象被local static对象替换掉。

这个手法的基础在于：C++保证函数内的local static对象会在“该函数被调用期间”“首次遇上该对象之定义式”时被初始化。也就是说以“函数调用”替换“直接访问non-local static对象”，也就能保证所获得的reference将指向一个经历初始化的对象。而且，如果从未调用non-local static对象的“仿真函数”，就不会引发构造和析构成本。

  • 为内置型对象进行手工初始化，因为C++不保证它们的初始化。

7.为多态基类声明virtual析构函数

当我们使用一个derived class的对象，但这个对象却是由base class指针调用析构删除的，就会引发未定义行为。通常结果是该对象的derived成分没有被删除，这会导致一个局部销毁的情况出现，从而引发内存泄漏。

所以进行基类声明时，需要将他的析构函数定义为virtual的。如果一个class不含virtual函数，通常表示他并不意图作为一个base class。 对于不作为base class的析构函数而言，不应该将它设定为virtual的。因为含有virtual的函数会导致对象带有一个虚函数表，导致额外的内存开销。

心得：只有当class内至含有一个virtual函数时，才会将析构函数声明为virtual的析构函数。

也不应该将STL的容器作为基类进行继承，因为他们的析构函数都是non-virtual的。

希望获得抽象类，但又没有纯虚函数时：可以将抽象类的析构函数定义为纯虚函数，并为其提供定义。

纯虚析构函数保证这个类为一个抽象类，不会被实例化，然而必须为这个纯虚析构函数提供定义，不然连接器会出问题。因为cpp中派生类的析构函数是首先调用最深派生类的析构函数，之后层层调用直到基类的析构函数，所以我们必须提供定义。

8.析构函数内不可抛出异常

析构函数内不要抛出异常，不然比如vector<T> v中的T是一个允许析构函数抛出异常的类。则会出现v结束生命周期时，一次调用v中元素的析构函数，如果只有一个抛出异常，其余元素依然应该被销毁。但如果其余元素中又有一个在析构过程中抛出异常，则出现了两个同时作用的异常，这对于cpp来说是未定义行为，所以不应该在析构函数中抛出异常。

当使用new时，有两件事发生：

  1. 内存被分配出来
  2. 针对此内存会有一个（或多个）构造函数被调用

同样delete也会有两件事发生：

  1. 针对此内存的一个（或多个）析构函数被调用
  2. 内存被释放

delete的最大问题在于：即将被删除的内存之内究竟有多少对象，这决定了有多少个析构函数被调用。

对于一个非数组对象调用delete []会导致编译器将内存中某一部分作为数组长度读取，错误的调用析构函数。

由此引出另一个问题，就是typedef，例如typedef string AddressLines[4]，则delete时需要记得使用delete[]

21必须返回对象时，就返回对象

当需要返回一个对象时，不要尝试：

  1. 返回一个pointer或reference指向一个local stack对象（对象在语句块结束后被销毁）
  2. 返回一个reference to a heap-allocated object（无法成功析构）
  3. 返回一个pointer或reference指向一个local static对象，而有可能同时需要多个这样的对象。（同时需要多个这种对象时，引用会导致他们永远指向同一个对象，比如func(a, b) == func(c, d)永远成立，func(a, b)返回一个local static object的指针或引用）
